# Plan

## Summary

Add failsafe validation to reject Java `DateTimeFormatter` optional-section brackets (`[` and `]`) from the internal date format string used during translation. HubL's `format_datetime` filter (and Apache Velocity's `$time.input`) use Java `SimpleDateFormat`, which does not support bracket-based optional sections ŌĆö only `DateTimeFormatter` does. The current implementation is incidentally safe (the default `DEFAULT_INTERNAL_FORMAT` has no brackets, and the PHP format string whitelist already blocks them at parse time), but the `internal_format` translation parameter accepts arbitrary strings with no validation, creating a risk surface.

## Architectural Context

The date translation pipeline has three format-string touch points:

1. **PHP format string** (user-facing, e.g. `d/m/Y`): Validated character-by-character by `Mailcode_Date_FormatInfo::validateFormat()` in [src/Mailcode/Date/FormatInfo.php](src/Mailcode/Date/FormatInfo.php#L142-L175) against a closed whitelist. Brackets are already rejected here.

2. **Output LDML/Java format** (e.g. `dd/MM/yyyy`): Generated by a 1:1 character mapping from the validated PHP format via `$charTable` in [src/Mailcode/Translator/Syntax/ApacheVelocity/ShowDateTranslation.php](src/Mailcode/Translator/Syntax/ApacheVelocity/ShowDateTranslation.php#L47-L68). No brackets can appear because no mapping produces them.

3. **Internal format** (Java format string, e.g. `yyyy-MM-dd'T'HH:mm:ss.SSSXXX`): Used only by Apache Velocity's `ShowDateTranslation::getInternalFormat()` ([line 71](src/Mailcode/Translator/Syntax/ApacheVelocity/ShowDateTranslation.php#L71-L80)). This is embedded directly in the output as `time.input("FORMAT", $var)`. It defaults to the safe `DEFAULT_INTERNAL_FORMAT` constant but can be overridden via `$command->setTranslationParam('internal_format', $value)` ŌĆö **with no validation**.

HubL's `ShowDateTranslation` ([src/Mailcode/Translator/Syntax/HubL/ShowDateTranslation.php](src/Mailcode/Translator/Syntax/HubL/ShowDateTranslation.php)) does **not** currently consume `internal_format`. However, the validation should be proactive to protect both syntaxes.

Key classes involved:
- `Mailcode_Date_FormatInfo` ŌĆö PHP format validation ([src/Mailcode/Date/FormatInfo.php](src/Mailcode/Date/FormatInfo.php))
- `ApacheVelocity\ShowDateTranslation` ŌĆö Consumes `internal_format` ([src/Mailcode/Translator/Syntax/ApacheVelocity/ShowDateTranslation.php](src/Mailcode/Translator/Syntax/ApacheVelocity/ShowDateTranslation.php))
- `HubL\ShowDateTranslation` ŌĆö Date translation for HubL ([src/Mailcode/Translator/Syntax/HubL/ShowDateTranslation.php](src/Mailcode/Translator/Syntax/HubL/ShowDateTranslation.php))
- `Mailcode_Commands_Command` ŌĆö Generic `setTranslationParam()`/`getTranslationParam()` ([src/Mailcode/Commands/Command.php](src/Mailcode/Commands/Command.php#L513-L522))

## Approach / Architecture

Add a static validation method to `Mailcode_Date_FormatInfo` that checks a Java format string for unsupported optional-section brackets. Call this method from `ApacheVelocity\ShowDateTranslation::getInternalFormat()` after resolving the format value, throwing a `Mailcode_Translator_Exception` if brackets are found.

The validation lives in `FormatInfo` (not in the translator) because it is a date-format concern, and this centralizes it for reuse by any future syntax that consumes an internal format (e.g., if HubL ever adds one).

A simple bracket-character check (`[` or `]`) is sufficient because:
- These are the only characters that denote optional sections in `DateTimeFormatter`.
- `SimpleDateFormat` (used by both target platforms) has no valid use for square brackets.
- A full Java format parser would be over-engineering for a failsafe guard.

## Rationale

- **Why not validate in `setTranslationParam()`?** That method is generic (any key/value), and the validation is specific to the `internal_format` parameter. Validating at the point of consumption follows the existing project pattern (e.g., `FormatInfo::validateFormat()` is called by the command, not by the parameter setter).

- **Why not a full Java format validator?** The only problematic syntax is brackets. Other Java format characters (`y`, `M`, `d`, `H`, `m`, `s`, `S`, `X`, `'`, etc.) are all valid in `SimpleDateFormat`. A bracket check is the right granularity.

- **Why add to `FormatInfo` instead of the translator?** Centralizes all date-format validation in one class, consistent with the existing `validateFormat()` method for PHP format strings. Any future syntax can reuse it.

## Detailed Steps

1. **Add `validateJavaFormat()` to `Mailcode_Date_FormatInfo`.**
   - New public static method: `validateJavaFormat(string $formatString): OperationResult`.
   - Check for presence of `[` or `]` in the string.
   - Return an `OperationResult` with error if brackets are found, success otherwise.
   - Add a new validation error code constant (e.g. `VALIDATION_JAVA_OPTIONAL_BRACKETS_NOT_SUPPORTED = 55803`).

2. **Call the validation in `ApacheVelocity\ShowDateTranslation::getInternalFormat()`.**
   - After resolving the format string (either from translation param or default), call `Mailcode_Date_FormatInfo::validateJavaFormat()`.
   - If the result is an error, throw `Mailcode_Translator_Exception` with a descriptive message and a new error code (e.g. `ERROR_INTERNAL_FORMAT_CONTAINS_OPTIONAL_BRACKETS = 55503`).

3. **Add test cases.**
   - In `tests/testsuites/Translator/Velocity/ShowDateTests.php`: Add a test that sets `internal_format` to a bracketed value (e.g. `yyyy-MM-dd['T'HH:mm:ss]`) and asserts that translation throws the expected exception.
   - Optionally: Add a unit test for `Mailcode_Date_FormatInfo::validateJavaFormat()` directly.

4. **Update documentation.**
   - Add a note to `docs/user-guide/translate-apache-velocity.md` in the internal format section noting that optional brackets are not supported.
   - Add a note to `docs/agents/project-manifest/constraints.md` under Date translation constraints (or HubL coverage) noting this validation.
   - Update `docs/agents/project-manifest/api-surface.md` with the new `validateJavaFormat()` method signature.

## Assumptions

- The `internal_format` translation parameter is the only unvalidated Java format string that enters the translation pipeline. The output format and PHP format strings are already protected by existing validation.
- HubL does not currently consume `internal_format`, so no changes to the HubL translator are needed now. If HubL adds internal format support in the future, it should call the same `validateJavaFormat()` method.
- The `DEFAULT_INTERNAL_FORMAT` constant is safe (no brackets) and does not need to change.

## Constraints

- PHPStan level 9 compliance must be maintained.
- `declare(strict_types=1)` in all modified files.
- Error codes must be unique across the codebase.

## Out of Scope

- Full Java SimpleDateFormat parser/validator (over-engineering for a failsafe).
- Modifying `setTranslationParam()` to add parameter-specific validation (generic method, wrong layer).
- Adding `internal_format` support to HubL (separate feature, not requested).
- Validating the output LDML format string (already protected by the character mapping table).

## Acceptance Criteria

- Setting `internal_format` to a value containing `[` or `]` causes translation to throw `Mailcode_Translator_Exception` with a descriptive message.
- The default `DEFAULT_INTERNAL_FORMAT` continues to work without errors.
- Custom `internal_format` values without brackets (e.g. `yyyy-MM-dd`) continue to work.
- `Mailcode_Date_FormatInfo::validateJavaFormat()` returns an error `OperationResult` for bracketed strings and a success result for clean strings.
- All existing tests pass (no regressions).
- PHPStan analysis passes at level 9.

## Testing Strategy

- **Unit test for `validateJavaFormat()`**: Test with clean format strings (expect success) and format strings containing `[` and/or `]` at various positions (expect error).
- **Integration test for Apache Velocity translation**: Set `internal_format` to a bracketed value via `setTranslationParam()`, call `translateCommand()`, and assert the expected exception type and code.
- **Regression test**: Run the full existing `ShowDateTests` suite for both Apache Velocity and HubL to confirm no changes to existing behavior.

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| **Bracket check is too narrow** ŌĆö other invalid characters might exist in custom `internal_format` values | SimpleDateFormat accepts a very broad character set; brackets are the only clearly invalid syntax from DateTimeFormatter. Additional validation can be added incrementally if needed. |
| **Breaking existing consumers** ŌĆö someone already uses a bracketed `internal_format` | This is precisely the bug we want to catch. The brackets would produce invalid output in both target platforms; catching this early is the correct behavior. |
| **Future HubL `internal_format` support** needs separate implementation | The validation method is centralized in `FormatInfo`; any future HubL internal format support just needs to call it. |
